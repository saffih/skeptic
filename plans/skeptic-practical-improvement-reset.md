| ID | Finding | Evidence and impact | Corrective principle |

| RC5 | Rejected as a root cause: weak branch isolation | The benchmark branch stayed separate and `main` advanced independently. The coupling was procedural, not a Git isolation failure. | Preserve isolated incomplete branches and allow unrelated branches to proceed. |
| RC6 | Supported hypothesis: missing effective stop-loss | The unfinished pipeline and reset request are consistent with ineffective stop-loss behavior. Public records contain stage stops but do not establish the full retry history or cost. | Two failures of one class trigger task-design review, not another larger retry. |
| RC7 | Supported hypothesis: missing decision-value gate | Stage 4 was ceiling-limited and Stage 5E could not authorize promotion, while its public next step was private scoring. RC4 is the directly confirmed root cause. RC5 is rejected as causal. RC1, RC2, RC3, RC6, RC7, and RC8 remain supported working diagnoses: they justify prospective safeguards but are not asserted as proven history. No other root-cause candidate was rejected. The rejected RC5 diagnosis remains useful evidence that parking is reversible and does not require branch repair.

## Selected Meta-Plan

Select Plan B. It has the highest near-term decision value, lowest irreversible cost, and a clear falsification path. It preserves useful evidence without treating sunk cost as a reason to continue.

The bounded repair is an explicit checkpoint: the project owner reviews Stage 6E when the scope and risk-weighted coverage audit closes. Resume only if a live product decision could materially change from the private comparison and the resume gates below pass. Otherwise keep it parked. This event-based checkpoint avoids a recurring governance process.

Plan A is parked because its result remains exploratory and does not justify current critical-path cost. Plan C is parked because infrastructure without a live decision is the wrong constraint. Plan D is rejected now because no repeated material core gap has been established.

## Stop-Loss Policy

1. Two failures from the same environment or protocol class: stop execution and review the task design.
2. Custody or infrastructure work exceeds the effort spent on the decision: park the workstream unless the decision is critical.
3. A prompt grows materially after each failure: replace it with a smaller redesigned task rather than another amendment.
4. A task cannot name the product decision it unlocks: do not execute it.
5. Work safely isolated on its own branch may be parked without blocking unrelated work.

## Decision Gates

### Before Changing Skeptic Core

- Define Skeptic's target scope.
- Identify a material mechanism gap.
- Test the current baseline first.
- Reproduce the miss across appropriate cases.
- Consider deterministic, domain-specific, escalation, and no-change alternatives.
- Assess false-positive, context, composition, and maintenance costs.

### Before Benchmark-Infrastructure Work

- Name the live product decision requiring it.
- Replace agent ceremony with deterministic tooling where possible.
- Keep private evidence local.
- Make state durable across handoffs.
- Require a protocol smaller than the experiment it protects.

### Before Resuming Stage 6E

- Name the exact decision it unlocks.
- Confirm C0, C1, and C2 remain relevant.
- Verify private evidence without moving or exposing it.
- Define a small deterministic execution plan.
- Show expected decision value exceeds execution cost.
